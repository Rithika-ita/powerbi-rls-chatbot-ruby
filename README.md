# Power BI RLS Chatbot — Accelerator

A **reusable accelerator** that embeds a **Power BI report** side-by-side with an
**AI chatbot**, both enforcing **Row-Level Security (RLS)** — designed for the
"App Owns Data" pattern where end users are **not** authenticated against
Entra ID.

```
┌──────────────────────────────────────────────────────────┐
│  Browser                                                 │
│  ┌──────────────────────┐  ┌──────────────────────────┐  │
│  │  Embedded PBI Report │  │  AI Data Chatbot         │  │
│  │  (RLS-filtered)      │  │  "What were my sales?"   │  │
│  │                      │  │  → DAX generated by LLM  │  │
│  │                      │  │  → Executed server-side   │  │
│  │                      │  │  → RLS-filtered answer   │  │
│  └──────────────────────┘  └──────────┬───────────────┘  │
│             │ Embed Token             │ REST API calls    │
└─────────────┼─────────────────────────┼──────────────────┘
              │                         │
              ▼                         ▼
   ┌────────────────────┐   ┌────────────────────┐
   │ Embed Token        │   │   Sinatra Backend   │
   │ (EffectiveIdentity │   │  ─ DAX generation   │
   │  + RLS role)       │   │    (Azure OpenAI)   │
   └────────┬───────────┘   │  ─ DAX execution    │
            │               │    (master user /    │
            ▼               │     DefaultAzureCred)│
            ▼               │  ─ RLS via DAX       │
   ┌────────────────────┐   │    (CALCULATETABLE   │
   │  Power BI /        │   │     user filter)    │
   │  Fabric Service    │   └─────────┬───────────┘
   │ (RLS engine for    │             │
   │  report visuals)   │◄────────────┘
   └────────────────────┘
```

**Key architecture:**
- **Embedded report** — RLS enforced by the Power BI engine via embed tokens
  with `EffectiveIdentity`. Users see only their data in the report visuals.
- **Chatbot DAX queries** — Executed server-side via the `executeQueries` REST
  API using a delegated token (master user ROPC or Azure CLI delegated token,
  configured via `DAX_AUTH_MODE`). The LLM adds user-specific `CALCULATETABLE`
  filters based on `rls_config.json`, ensuring each user's chat answers match
  their RLS-scoped view.

## Demo Scope (Important)

This repository is a **PoC accelerator** and intentionally uses a simulated user
selector (`DEMO_USERS`) instead of real app authentication.

Before using these patterns in production:

- Replace the simulated user dropdown with real authentication (OIDC/OAuth/JWT).
- Derive the RLS identity from trusted server-side auth context, not client input.
- If your external auth subject is not the same as the Power BI RLS username,
  implement a server-side mapping in your authentication middleware.
- Keep the embed-token endpoint protected behind authenticated/authorized APIs.
- Add API rate limiting, audit logging, and security headers/CSP.

## Features

| Feature | Details |
|---------|---------|
| **Embedded Report** | Power BI JS SDK, "App Owns Data" pattern |
| **Row-Level Security** | Embed token carries `EffectiveIdentity` — enforced by PBI engine for both report _and_ chatbot queries |
| **AI Chatbot** | Azure OpenAI generates DAX from natural language via **function calling** |
| **Server-side DAX** | `executeQueries` REST API with delegated token; LLM adds user-specific CALCULATETABLE filters from `rls_config.json` |
| **Three-phase chat** | Phase 1: LLM generates DAX → Phase 2: server executes → Phase 3: LLM summarises results |
| **Dual auth** | SP (client credentials) for embed tokens; master user (ROPC) or Azure CLI delegated token for DAX execution — configurable via `DAX_AUTH_MODE` |
| **User simulation** | Dropdown to switch between demo users and see different RLS views |

## Quick Start

### 1. Clone & install

```bash
cd powerbi-rls-chatbot-ruby
bundle install
```

### 2. Configure environment

```bash
cp .env.example .env
```

Edit `.env` and fill in your values:

| Section | Variables | Where to find them |
|---------|-----------|-------------------|
| **Service Principal** | `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET` | Entra ID → App Registrations |
| **Power BI** | `PBI_WORKSPACE_ID`, `PBI_REPORT_ID`, `PBI_DATASET_ID`, `PBI_RLS_ROLE` | Power BI service URL or REST API |
| **Azure OpenAI** | `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_DEPLOYMENT`, etc. | Azure Portal → OpenAI resource |
| **App** | `APP_SECRET_KEY`, `DEMO_USERS` | Generate a random key; set demo user map |

> **Tip:** If `AZURE_OPENAI_API_KEY` is set it will be used directly (API key auth).
> If blank, this Ruby app falls back to Azure CLI bearer token (`az login`)
> for Azure OpenAI requests.

### 3. Generate schema (first-time setup)

```bash
ruby setup.rb
```

This discovers your dataset schema and writes `sample_report/schema.json`,
which the LLM uses to generate correct DAX queries. Re-run whenever your
Power BI model changes.

> You can also create `sample_report/schema.json` manually — see
> [Schema File](#schema-file) below.

### 4. Run

```bash
ruby app.rb -o 0.0.0.0 -p 4567
```

Open **http://localhost:4567** — select a demo user from the dropdown.

## How It Works

### Report Embedding with RLS

```
Demo mode:
User selects "Alice" → POST /api/embed-token { rls_username: "alice@contoso.com" }
  → Server acquires SP token (client credentials)
  → Server validates identity against DEMO_USERS allowlist
  → Calls PBI GenerateToken with EffectiveIdentity(username=alice, role=ViewerRole)
  → Embed token returned → PBI JS SDK renders report filtered by RLS

Authenticated mode:
Browser calls POST /api/embed-token {}
  → Server resolves external auth subject from trusted auth context
    (for example from your Rack middleware / session)
  → Optionally maps subject to RLS username in server code
  → Calls PBI GenerateToken with EffectiveIdentity(mapped RLS username)
```

### Chat with RLS (three-phase flow)

```
Alice asks "What were total sales by region?"

  Phase 1 — Generate DAX
  → POST /api/chat/generate-dax { message, history, rls_username }
  → Server sends user message + dataset schema + rls_config to Azure OpenAI
  → LLM uses function calling to produce DAX with user filter:
      EVALUATE
      CALCULATETABLE(
          SUMMARIZECOLUMNS('SalesTerritories'[SalesTerritoryRegion],
                           "Total", SUM('Sales'[SalesAmount])),
          'Users'[UPN] = "alice@contoso.com"
      )

  Phase 2 — Execute server-side
  → POST /api/chat/execute-dax { dax }
  → Server calls Power BI executeQueries REST API (delegated token)
  → CALCULATETABLE filter ensures Alice only sees her region's data
  → Results: [{ SalesTerritoryRegion: "West", Total: 4050 }]

  Phase 3 — Summarise
  → POST /api/chat/summarize { message, dax, results, history }
  → LLM formats results: "Your total sales were $4,050, all in the West region."
```

The LLM always wraps data queries in `CALCULATETABLE` with a filter on the
identity table (e.g. `'Users'[UPN] = "alice@contoso.com"`), ensuring each
user's chat answers match their RLS-scoped view of the embedded report.

## Schema File

**`sample_report/schema.json`** tells the LLM what tables, columns, and
measures exist so it can write correct DAX.

```json
{
  "tables": [
    {
      "name": "Sales",
      "columns": ["ProductKey", "SalesAmount", "OrderDate", "Region"],
      "measures": ["Total Sales", "YoY Growth %"]
    },
    {
      "name": "Products",
      "columns": ["ProductKey", "ProductName", "Category"],
      "measures": []
    }
  ]
}
```

`setup.rb` auto-discovers this via DAX queries (`COLUMNSTATISTICS()` and
`INFO.MEASURES()`) and filters out hidden system tables. You can also create
or edit it manually.

> **Tip:** Re-run `setup.rb` whenever your Power BI model changes (new tables,
> renamed columns, etc.) to keep the schema in sync.

## Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| Embed token 403 | SP not added to workspace, or wrong workspace/report ID | Add SP as Member+ in PBI workspace settings |
| DAX execution 401 | `executeQueries` tenant setting disabled, or delegated token expired/invalid | Enable "Dataset Execute Queries REST API" in Fabric Admin Portal; check `DAX_AUTH_MODE` config; for `master_user` verify the account has no MFA and has a Pro/PPU license |
| Chat returns empty results | User has no data for their RLS role | Expected — the LLM will note "no data found" |
| LLM generates invalid DAX | Schema file is out of date or incomplete | Re-run `setup.rb` or update `schema.json` manually |
| LLM omits user filter | `rls_config.json` missing or `enabled: false` | Re-run `setup.rb` to generate `rls_config.json` |
| OpenAI 401 | API key wrong or Azure CLI session missing | Set `AZURE_OPENAI_API_KEY` or run `az login` |

## Security Notes

- **Service principal credentials never reach the browser.** All PBI API calls
  are server-side; the browser only receives the scoped embed token.
- **The LLM cannot bypass RLS.** The embedded report uses EffectiveIdentity
  in the embed token — the Power BI engine enforces RLS on all visuals.
  For chat DAX queries, the LLM is instructed to always include user-specific
  `CALCULATETABLE` filters based on `rls_config.json`. While this is
  LLM-enforced (not engine-enforced), the filter pattern is injected by the
  server's system prompt — not controllable by end users.
- **No stored passwords.** Auth uses Service Principal client credentials
  (client secret) for embed tokens. If `DAX_AUTH_MODE=master_user`, the
  master user password is stored in `.env` (server-side only, never sent
  to the browser). Keep `.env` out of source control.
- **Map users securely.** Replace the demo dropdown with your auth system
  (JWT, OAuth2, session) in production.

## Production Considerations

| Area | Recommendation |
|------|----------------|
| **Authentication** | Replace demo dropdown with your auth system (JWT, OAuth, etc.) |
| **Token caching** | MSAL handles SP and master-user token caching; consider caching embed tokens (TTL ~1 hr) |
| **DAX auth** | Set `DAX_AUTH_MODE=master_user` with a dedicated service account; or use Managed Identity via `default_credential` if the MI has a Pro license |
| **Rate limiting** | Add server-side rate limits on chat endpoints |
| **Schema refresh** | Re-run `setup.rb` when the Power BI model changes |
| **Streaming** | Add SSE/WebSocket for streaming LLM responses |
| **Monitoring** | Application Insights or equivalent |
